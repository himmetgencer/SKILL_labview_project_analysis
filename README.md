## Prerequisites

This skill requires two LabVIEW tools to generate the input files:

### 1. PlantUML Output — classy Diagram Viewer
**Source:** https://gitlab.com/tatiana.boye/classy-diagram-viewer

classy is a VI Scripting tool that generates PlantUML class diagrams from your LabVIEW project.
Unlike the built-in Class Hierarchy viewer, it captures inheritance, composition, and aggregation
relationships, including method signatures and data members.

**How to generate:**
1. Open your `.lvproj` in LabVIEW
2. Run classy and point it to your project
3. Export as PlantUML text (`.txt`)
4. Feed the text into this skill

### 2. Antidoc HTML Output — Wovalab Antidoc
**Source:** https://www.vipm.io/package/wovalab_lib_antidoc/

Antidoc generates structured documentation from your LabVIEW project's VI descriptions,
class documentation, and library documentation. Output formats: AsciiDoc, HTML, PDF.

**How to generate:**
1. Install via VIPM: `wovalab_lib_antidoc` (LabVIEW 2020+)
2. Open Antidoc from the Tools menu
3. Select your `.lvproj`
4. Export as HTML
5. Feed the HTML into this skill

> **Note:** Documentation quality directly affects analysis depth.
> Populate VI descriptions (File > VI Properties > Description),
> class descriptions, and library descriptions before running Antidoc.
> Follow the DQMH Consortium Style Guide for consistent results.

## INPUT FORMATS

### 1. PlantUML Text (plant-uml.txt)
Format generated by the classy Diagram Viewer tool.

**How to read:**
```
"ClassName"                         → lvclass name
{field} -VarName : Type             → private data member
{field} +VarName : Type             → public data member
{method} +MethodName()              → public VI (dynamic dispatch or override)
{method} #MethodName()              → protected VI (override only)
{static} {method} +MethodName()    → static VI (non-dynamic dispatch)
"Parent" <|-- "Child"              → inheritance (Child extends Parent)
"ClassA" *-- "ClassB"              → composition (ClassA contains ClassB)
"ClassA" o-- "ClassB"              → aggregation
namespace "LSL.lvlib:"             → lvlib namespace
```

**Actor Framework-specific patterns:**
```
"XxxActor.lvclass"                       → actor inheriting from Actor.lvclass
"Send Xxx Msg.vi"                        → message sender VI (command message)
"Do.vi"                                  → VI executed inside Actor Core
"Actor Core.vi" override                 → actor's main loop
"Pre-Launch Init.vi" override            → pre-launch resource initialization
"Handle Error.vi" override               → error handling
```

**DQMH-specific patterns:**
```
"XxxModule.lvlib:"                       → DQMH module
"Main.vi"                                → module's main VI (EHL + MHL)
"Helper Loop.vi"                         → for async operations
"Broadcast: Xxx.vi"                      → outbound event
"Request: Xxx.vi"                        → inbound request (fire & forget)
"Request and Wait for Reply: Xxx.vi"     → synchronous request-reply
"Synchronize Module Events.vi"           → event synchronization
```

**JKI State Machine-specific patterns:**
```
Single-VI QSM architecture
State strings: "Init", "Main", "Error", "Stop"
SubState syntax: "SubVI >> NextState"
```

## ANALYSIS FRAMEWORK

### Layer 1: Architecture Analysis (from PlantUML)

```
Answer the following:
1. Which framework(s) are used? (AF / DQMH / JKI / Hybrid / Custom)
2. What is the class hierarchy? (inheritance tree)
3. What are the actor/module responsibilities?
4. What is the messaging pattern? (command / broadcast / request-reply)
5. What are the composition relationships? (which class contains which)
6. Are interfaces used?
7. Are design patterns present? (Singleton, Factory, Observer, etc.)
```

### Layer 2: Behavioral Analysis (from Antidoc)

```
Answer the following:
1. What is each module/actor responsible for?
2. How does data flow? (which VI feeds which)
3. How is error handling implemented?
4. What are the state machine states?
5. How is hardware abstraction structured?
6. What is the threading/parallelism strategy?
```

### Layer 3: Quality Analysis (against Style Guide)

```
Check against the DQMH Style Guide:
□ VI Documentation populated?
□ Class Documentation populated?
□ Connector pane using 4-2-2-4 pattern?
□ Error handling correct?
□ Typedef usage appropriate?
□ Naming convention (Spaced Pascal Case) correct?
□ File organization consistent?
```

## FRAMEWORK-SPECIFIC KNOWLEDGE

### Actor Framework (AF)

```
Core concepts:
- Actor: Independently running object with a message queue
- Root Actor: Application entry point
- Nested Actor: Actor launched by another actor
- Message Class: Separate .lvclass per message type
  - Do.vi: Code executed inside Actor Core for this message
  - Send Xxx Msg.vi: Static VI that enqueues the message
- Actor Core.vi: Main loop (must be overridden)
- Pre-Launch Init.vi: Queue and resource initialization
- Handle Error.vi: Error handling

Architectural patterns:
- Parent-Child: Parent launches and monitors child
- Sibling: Two children of the same parent message each other
- AutoStop: When child stops, parent stops too
- Nested Endpoint Actor (2.0+): Actors across different executables

Common issues:
- Deadlock: Circular wait (A waits for B; B waits for A)
- Message flooding: Queue overflow
- AutoStop cascade: Unexpected shutdown chain
- Memory leak: Actor references not cleaned up
```

### DQMH (Delacor Queued Message Handler)

```
Core concepts:
- Module: Independently running unit (.lvlib)
- Main.vi: EHL (Event Handling Loop) + MHL (Message Handling Loop)
- Helper Loop: Additional loop for async tasks
- Broadcast Event: Event published outward by the module
- Request Event: Inbound request (fire & forget)
- Request and Wait: Synchronous request-reply

Scripting tools:
- Add New Module
- Add New Request
- Add New Broadcast
- Rename Module
- Validate Module

DQMH vs AF:
- DQMH: More accessible, OOP not required, EHL runs on UI thread
- AF: Full OOP, extensible via inheritance, more complex

Version matters: DQMH 7.x differs significantly from earlier versions!
```

### JKI State Machine (JKISM)

```
Pattern: String-driven state machine in a single VI
State syntax: "StateName"
SubVI call: "SubVI Name >> NextState"
Error handling: "Error" state
Initialization: "Init" state

Typically used for simple modules or utility VIs.
In large projects, appears as a small sub-component alongside AF or DQMH.
```

### Hybrid Architectures (LabVIEW + Python / C# / ReactJS)

```
Common integration patterns:

1. LabVIEW ↔ Python:
   - Python Node (LabVIEW 2018+): Direct Python calls
   - TCP/IP: Socket-based messaging
   - Shared Memory: High-performance data transfer
   - File-based: CSV/JSON sharing (simple but slow)

2. LabVIEW ↔ C#:
   - .NET Assembly: Using C# DLL from LabVIEW
   - LabVIEW Web Services: REST API
   - ActiveX/COM: Legacy integration

3. LabVIEW ↔ ReactJS:
   - SystemLink or custom Web Services
   - WebSocket: Real-time data streaming
   - REST API: Request-response
   - LabVIEW NXG WebVIs (if applicable)

Analysis cues:
- .NET or Python Node visible in PlantUML → hybrid integration present
- "Web Service" or "HTTP" visible → frontend integration present
```

## ANALYSIS PROCEDURE

When the user provides PlantUML + Antidoc, analyze in this order:

### Step 1: Quick Inventory
```
- How many classes? How many lvlibs?
- Which framework(s) are used?
- What is the project's purpose? (test system, control system, data acquisition, etc.)
- Is there a hybrid stack?
```

### Step 2: Architecture Map
```
List main actors/modules:
[Actor/Module Name] → [Responsibility] → [Connected to]

Show hierarchy:
RootActor
├── ActorA (responsibility)
│   └── ActorA1 (responsibility)
└── ActorB (responsibility)
```

### Step 3: Messaging Flow
```
Trace critical data/command flows:
"Command X → Actor A → Actor B → hardware"
"Sensor data → Actor C → broadcast → Modules D, E"
```

### Step 4: Points of Concern
```
Flag potential issues:
- Deadlock-prone points
- Thread safety risks
- Style Guide violations
- Undocumented critical VIs
```

## QUESTION TYPES & RESPONSE STRATEGIES

### Bug Fix
```
"X doesn't work / Y throws an error"

Strategy:
1. Locate the relevant VI in Antidoc
2. Check class relationships in PlantUML
3. Trace the message flow
4. List likely causes (framework-aware)
5. Suggest a specific VI/wire/method fix

Common AF bugs: null reference in message Do.vi
Common DQMH bugs: Synchronize Module Events not called
Common JKISM bugs: SubVI >> NextState syntax error
```

### Architecture Question
```
"How should I design / extend this module?"

Strategy:
1. Propose something consistent with the existing hierarchy
2. Follow framework conventions
3. Reference the DQMH Style Guide
4. Identify reuse opportunities (inherit from existing classes)
5. Suggest concrete VI names and class structure
```

### Migration Question
```
"Migrate this LabVIEW code to Python / C#"

Strategy:
1. Identify the algorithm logic (strip framework overhead)
2. Map data types (LabVIEW → target language)
3. Map concurrency model (actor/loop → thread/async)
4. Identify the hardware abstraction layer (mock needed?)
5. Provide a step-by-step migration plan
```

### Code Review
```
"Review this code"

Strategy:
1. Apply Style Guide checklist
2. Check framework best practices
3. Flag performance concerns
4. Highlight documentation gaps
5. Suggest concrete fixes (LabVIEW-specific)
```

## IMPORTANT NOTES

1. **PlantUML limitation**: PlantUML shows inheritance and composition only.
   Wire connections and block diagram internals are not visible. State this when making inferences.

2. **Antidoc limitation**: Only shows documented items.
   For undocumented VIs, flag "documentation missing."

3. **Framework version differences**:
   - AF 2.0+ vs earlier (Register Actor Msg removed, auto registration added)
   - DQMH 7.x vs 6.x (significant API changes)
   - LabVIEW version matters (2019 vs 2024)

4. **Hybrid projects**: For code outside the LabVIEW boundary,
   apply the best practices of the relevant technology (Python / C# / React).

5. **The user is an expert**: Do not state basic concepts.
   Never write "In Actor Framework, every message is a class."
   Instead say directly: "In this pattern, overriding Xxx.vi would be cleaner."

## EXAMPLE ANALYSIS: plant-uml.txt (LSL Project)

The provided `plant-uml.txt` is a **LabVIEW Scripting Language (LSL)** interpreter project.

```
Architecture: Custom OOP (not AF/DQMH — script engine)
Namespace: LSL.lvlib

Main class hierarchy:
functionbase (abstract base)
├── Array Functions
├── Bitwise Functions
├── Boolean Functions
├── Comparison Functions
├── Numeric Functions
├── Statement
├── String Functions
├── Structure If Else
├── Structure Loop  (additional fields: index, loop size)
├── Timing Functions
└── UI Functions

Output Terminal (abstract)
└── PainOutputTerminal

Composition:
functionbase *-- Output Terminal  (every function owns an output terminal)
functionbase *-- RT Variables     (runtime variable management)
scripts manager o-- functionbase  (aggregation: manages list of functions)

scripts manager → orchestrator: compiles and executes scripts
RT Variables → singleton: GetSet Singleton pattern present
```

**Points of concern:**
- `Structure Loop` has private `-index` and `-loop size` fields → nested loop state management
- `scripts manager` has a `Valid Script?` Boolean field → compile-time validation gate
- `Run_Init`, `Run_Pre`, `Run_Core`, `Run_Post`, `Run_Close` → lifecycle hooks (Template Method design pattern)
- `SmartSyntax` fields → custom syntax parsing layer
